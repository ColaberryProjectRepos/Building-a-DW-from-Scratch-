Strategy #1 - Find Transaction tables and determine relationships

If you read the notes in the OLTP vs. OLAP Discussion, you would know that you have transaction tables and descriptive tables in a database.  Your transaction tables are used to record a process such as going through the checkout line at Walmart.  Your descriptive tables will define that process.  How do you quickly find what the transaction tables are?  They are usually the biggest tables.  Not always, but more times than not.

If you run the below query on any database, it will list out all of the tables in the database and sort them by row count descending.

CREATE TABLE #counts
(
    table_name varchar(255),
    row_count int
)

EXEC sp_MSForEachTable @command1='INSERT #counts (table_name, row_count) SELECT ''?'', COUNT(*) FROM ?'
SELECT table_name, row_count FROM #counts ORDER BY row_count DESC
DROP TABLE #counts

Going through this process for all of the columns gives you a much better understanding of everything that is being tracked in this business process.  Now we want to look for dimensions to join.  These will usually be in the form of Foreign Keys (Either columns that end in ID or Key)  I have circled 2 above.

Let's do StatusID - We will want to find a Status table and join it to the Repairs Order table while keeping the existing filter on so we can see how the data reacts to the join.

There are 2 tables with Status in them



How do we know which one to use?  Now we have to use some good ole problem solving skills.  In the StatusCategory table, it has a CategoryKey as the Primary Key and StatusCodes has StatusID which matches our table.  That has me leaning towards StatusCodes as my table to join, but if you are still unsure, you can do other checks ... such as look at the distinct values of the StatusID column in both tables.  If you do a distinct on both tables with the StatusID columns, you will see if the values match.



I am now 100% sure that StatusCodes is my table so I join them with my original filter on.



Now I have to go through this record and ask myself does it make sense.  Each Repair Order should have a status.  My status here is "Not in Process". That makes sense to me.  Also it didn't create more than the 1 original record that I started with, so my join looks good.  Now I'm ready to repeat this process until I link up as many tables in the data model as I can. 

Repeat this process by 1st looking at all of the ID or Key columns in our transaction table (RepairOrders)

Then do the same process for each joining table.  See if they have ID's or Keys and find out what they join to.  Repeat this process until you run out of ID's or tables to join.  Your result should be a multi table join that returns one big dataset.  This ensures that all of the data is related and this is a crucial step in designing the DataMart/Warehouse.

Now it's time to scope out our Fact and Dimension tables.

Go through every column in your big dataset and categorize them.  You only want data that will help you report (valuable dimension and keyfacts)  and key columns for connecting.  Columns that you don't think add value to the reporting will not be included in this process.

Facts
Tables.Column  (Example: tblRepairOrders.TotalLoss)

Dimensions
DimensionName.Tables.Column  (Example: Status.tblStatusCodes.StatusDesc)

Now that you know what your aggregate and descriptive data will look like, it's time to start asking questions you think the business will want to know.  These question will ultimately drive the reporting on the back end.  Be very created because the better the questions, the more awesome the ideas you will have for your reporting.  Here are some examples.

1.  How many cars come in each shop
2.  What are most common repair problems
3.  What are the most expensive repair problems
4.  What is the average time it takes to get your car repaired and which shops are faster at different fixes.
